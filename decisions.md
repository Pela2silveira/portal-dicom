## Decisions
- MVP first, before auth features.
- The first deliverable must be a Docker Compose stack.
- The stack must include `orthanc`, `backend`, `postgres`, `nginx`, and `ohif`.
- Local Orthanc is mandatory and acts as the cache and retrieve target.
- PostgreSQL is the default persistence layer for operational data unless a clearly better fit is justified during the design discussion.
- Orthanc is the source of truth for cached studies; Postgres keeps only an operational index that can be rebuilt.
- No doctor login in the MVP.
- No patient login in the MVP.
- No email-code verification, sessions, JWT, or share links in the MVP.
- HIS integration is configuration-first in the MVP and patient lookup is not required to be fully operational in the first build slice.
- The first build slice follows HIS Option B: persist and validate HIS configuration only, without executing real Andes MPI patient lookups yet.
- As a temporary unblocker before the REST HIS integration is available, the backend may use a direct MongoDB read-only adapter to resolve patient identity data.
- This MongoDB integration is strictly backend-only and must not leak Mongo-specific concerns into handlers, UI contracts, or general domain logic.
- The temporary MongoDB adapter must be encapsulated behind the same identity-source abstraction that will later support the REST HIS provider.
- The temporary MongoDB adapter must be read-only: no inserts, updates, deletes, migrations, schema changes, or side effects are allowed through portal code.
- MongoDB queries used for patient lookup must be explicitly performance-oriented: indexed lookups only, bounded result sets, projection-limited reads, and no collection scans.
- The temporary MongoDB adapter is transitional and must remain replaceable by the future REST HIS integration without changing portal-facing contracts.
- The backend now owns a stable `PatientIdentitySource` abstraction so patient identity resolution remains decoupled from handlers and can switch from local seed to MongoDB or REST providers without changing patient-facing endpoints.
- When `his.provider = his_mongo_direct`, the backend resolves patient identity directly against MongoDB collection `paciente` in read-only mode and persists successful normalized results into Postgres.
- Patient remote QIDO search must use both the canonical document number and the persisted Mongo `_id` string (`mongo_object_id`) when available, merging matches by `StudyInstanceUID`.
- Patient remote QIDO search must fan out across all configured `qido_rs` PACS nodes instead of assuming a single remote node.
- Patient remote search may now also probe `c_find` / DIMSE nodes. The first authorization rule is strict: studies are exposed only when the remote `PatientID` matches one of the patient identifiers loaded from Andes/HIS, while identity-comparison signals continue to be logged for later matching rules.
- Patient DIMSE search now performs two discovery paths against compatible nodes: identifier-based probing using all known patient identifiers, and demographic probing using normalized birth date, normalized sex, and a Synapse-compatible surname wildcard query.
- Patient demographic DIMSE probing against Synapse must use a surname wildcard `PatientName` query such as `CASTRO*`; the previous full-name fuzzy formatting is not reliable for this interoperability path.
- Patient DIMSE authorization currently accepts studies either by strict `PatientID` match against portal-known identifiers or by a strong demographic match combining birth date, sex, and high-confidence patient-name correspondence.
- Background patient search workers must reload the full persisted `PatientSummary` before remote execution so DIMSE demographic probing always has access to `full_name`, `birth_date`, and `sex`.
- Professional remote search must stop assuming a single globally configured PACS; the physician UI now selects one remote online PACS explicitly, while `local_cache` remains a first-class searchable source backed by Orthanc QIDO.
- Each PACS node may now carry an optional `andes_organization_id` in config as PACS metadata, but the current Mongo `prestaciones` enrichment resolves studies canonically through `metadata.pacs-uid = StudyInstanceUID`.
- The ANDES `prestaciones` enrichment must be guarded by `his.prestaciones_enrichment_enabled`; the feature is opt-in and remains disabled by default so Mongo latency cannot penalize the main search flows unless explicitly enabled.
- QIDO result persistence must be modeled as a shared cache keyed by `study_instance_uid + source_node_id`; this entity is independent of patient/professional flows and can be reused by both.
- The same shared cache must persist resolved ANDES enrichment fields when available so future searches can reuse them without re-querying Mongo `prestaciones`.
- Physician DIMSE `C-FIND` results must persist through the same shared remote-study cache and recent-query path used by QIDO results; otherwise retrieve from physician DIMSE result rows is not reliable.
- Professional retrieve must preserve source-node provenance for studies discovered by DIMSE `C-FIND`; when the primary source field is absent, origin resolution may fall back to equivalent recent-query source data.

- (2026-05-31) Runtime log handling for this project follows "Option A": all containers keep logging to stdout (12-factor), and a host-side systemd streamer captures the project's `docker compose logs` into plain files under `app/logs/` on the remote (`/opt/andes/portalimagenes/app/logs/`) for easy, sudo-free access close to the project. Two units: `portal-logs.service` → `portal.log` (full firehose) and `portal-logs-audit.service` → `audit.log` (only lines tagged `"channel":"audit"`). Retention is handled by `logrotate` (`daily`, `rotate 95` ≈ 3 months, compressed, `copytruncate`). Because `docker compose logs -f` does not follow recreated containers, the streamers are restarted at the end of every `deploy`/`update`. All of this is provisioned from `Makefile.deploy.local` (local-only, gitignored) via `logging-install` / `logging-restart` / `logging-status` / `logging-uninstall`; the systemd units, stream scripts, and logrotate config are generated on the remote, not committed. This is project-scoped and complements (does not replace) any global `daemon.json` json-file rotation on the shared host.

- (2026-05-29) The backend (`app/backend/cmd/server/`) is split from a single ~12.5k-line `main.go` into multiple files within the **same `package main`**, grouped by domain (`server.go`, `app.go`, `config.go`, `patient.go`, `physician.go`, `search.go`, `retrieve.go`, `orthanc.go`, `andes.go`, `identity.go`, `health.go`, `share.go`, `email.go`, `migrations.go`, `util.go`). ## Physician Search: DNI vs PatientID (resolved + implemented)

### Operator decision (2026-07-06) that unblocked this
- **PACS nodes are always queried by DICOM `PatientID`; the value that `PatientID` corresponds to is owned by the HIS, not the portal.** The id↔dni correspondence is the HIS's responsibility. Per node, that `PatientID` may be the DNI, the Mongo `_id`, or another HIS-provider field.
- **Searching by ID is a direct DICOM query with no intermediaries.** The value the professional types is sent verbatim as `PatientID`. It may coincide with the DNI but is not the same concept.
- **Searching by DNI resolves against the local cache first** (`patient_identifiers`, the same table the patient-portal login flow populates), refreshing from Mongo/HIS when needed. The mapping rule is declared per node in config.

### Implemented design (v0.7.0)
- `PhysicianSearchFilters` gains `DocumentNumber` (DNI), independent from `PatientID` (direct DICOM ID). The handler parses `document_number` and `patient_id` separately.
- Each PACS node declares a mapping rule via `patient_id_source` in config: `"dni"` (default), `"mongo_id"`, or a future HIS-provider field (unknown values are accepted but currently resolve to no query, leaving a hook).
- **Search by ID (`patient_id`)** takes precedence and is sent as-is to the selected source — unchanged behavior.
- **Search by DNI (`document_number`)** resolves identifiers once (DNI + Mongo id) via `resolvePatientSearchIdentifiers`: local `patient_identifiers` cache first, then a live `ensurePatientRecordWithIdentity` (HIS/Mongo) only when the Mongo id is actually required and missing — which also refreshes the cache. `effectivePatientIDForNode(node.patient_id_source, ids)` then picks the value sent as `PatientID` for that node. If the required identifier is unknown, the node returns no results (logged `physician_search_dni_unmapped`) rather than sending a wrong value.
- **Local cache DNI search** probes the local Orthanc once per distinct mapped identifier across configured nodes (`candidateLocalCachePatientIDs`) and merges unique studies by `StudyInstanceUID`, because the local cache aggregates studies keyed by each source node's mapped `PatientID`.
- Reusing `patient_identifiers` means correlations learned by the patient portal are leveraged automatically, and DNI searches that hit the HIS write the mapping back for reuse. No new `identifier_type` or migration was required.
- Pure helpers (`normalizePatientIDSource`, `patientIDSourceNeedsMongo`, `effectivePatientIDForNode`, `candidateLocalCachePatientIDs`) are covered by Tier 1 tests in `physician_patient_id_test.go`.
- UI: the professional filter block now exposes a **DNI del paciente** field (mapped to `document_number`) and a separate **ID DICOM (directo)** field (mapped to `patient_id`).
- (v0.7.1) UI field order: in the 2-column patient-lookup grid the first row splits into `DNI | ID DICOM (directo)` and the `Nombre` field spans the full width on the second row (`.field-group-span-full` → `grid-column: 1 / -1`). This keeps the two short identifiers side by side and gives the name its own wide row.
- (v0.7.1) Static asset caching fixed: `/portal-assets/portal.js` and `portal.css` were served without `Cache-Control`, so after a deploy browsers kept a stale `portal.js` while loading the new `index.html`. A mismatched pair (new HTML with the ID field, old JS without its wiring) made the professional ID search send no `patient_id` at all, returning unfiltered results. Fix: nginx now sends `expires -1` (revalidate) for `/portal-assets/` and the HTML routes, and the asset URLs carry a `?v=<version>` query. `expires` is used instead of `add_header Cache-Control` so the inherited security headers are preserved.

### Deferred
- Other HIS-provider mappings beyond `dni`, `mongo_id` and `legacy_his` are not implemented yet; unknown `patient_id_source` values are accepted and leave the hook.

## Physician Search: legacy HIS (MSSQL) DNI→Codigo (implemented v0.8.0)

### Context
- The `hpnsyn` node (Hospital Provincial Neuquén / Synapse) keys studies by an internal HIS patient code (`Historias_Clinicas.Codigo`), not the DNI or Mongo id. This is the concrete "other HIS-provider field" case the `patient_id_source` enum left as a hook.
- The mapping DNI→Codigo lives in a legacy Microsoft SQL Server database (`Hospital.dbo.Historias_Clinicas`, filter `HC_Documento`). It applies **only** to `hpnsyn`.

### Design
- New `patient_id_source: "legacy_his"` (aliases `mssql_codigo`/`codigo`). When a professional searches by DNI against a node with this rule, the DNI is mapped to `Codigo` and sent as the DICOM `PatientID`. Direct ID search is unaffected.
- New optional top-level `legacy_his` config block: `host`, `port` (default 1433), `database` (default `Hospital`), `user`, `password_env` (default `LEGACY_HIS_PASSWORD`), `encrypt` (default `disable` — legacy internal server without a valid cert), and timeouts. Credentials never live in JSON: the password is read from the env var named by `password_env`.
- Client (`legacy_his.go`): pooled `database/sql` + `github.com/microsoft/go-mssqldb` (pinned `v1.7.2`, the last release compatible with Go 1.22). Parameterized query `SELECT TOP 1 [Codigo] FROM [<db>].[dbo].[Historias_Clinicas] WHERE HC_Documento = @dni`. Startup is resilient: a build/ping failure records a startup issue and leaves a nil client (legacy_his DNI searches just don't map), so a down MSSQL never blocks boot.
- Resolution reuses `patient_identifiers`: `resolveLegacyHISCode` reads the cached `legacy_his_code` identifier first and, on a miss, queries MSSQL live and **best-effort persists** the mapping (`source_system='legacy_his'`, `identifier_type='legacy_his_code'`). Persistence only attaches to an already-existing patient row (created by the patient portal flow); unknown patients are resolved live without creating spurious `patients` rows (avoids polluting `last_login_at`). No migration needed (existing table/columns).
- `resolvePatientSearchIdentifiers` now takes a `patientIdentifierNeeds{Mongo, LegacyHIS}` struct instead of a single `needMongo` bool, so each lookup (Mongo / MSSQL) only runs when a configured node's rule requires it. Local-cache DNI search aggregates the needs across all configured nodes.
- Pure helpers (`legacyCodigoToString`, `buildLegacyHISDSN`, `legacyHISLookupQuery`, `patientIDSourceNeedsLegacyHIS`, `patientIdentifierNeedsForSources`, `effectivePatientIDForNode` legacy case) covered by Tier 1 tests. The live MSSQL query itself is not unit-tested (no infra).
- **Health check (optional severity, never fatal).** When `legacy_his.enabled`, the system health collector adds a `legacy_his` component (`legacyHISComponent`) that pings the MSSQL. It is `ComponentSeverityOptional`, so an unreachable legacy HIS only marks the system `degraded` (like a remote PACS down) and can never make `/health` return `unavailable`/503 — the app must not go down because the legacy HIS is offline. Startup is likewise non-fatal: a failed build/ping records a startup issue and leaves a nil client (the component then reports optional/unavailable). This mirrors the existing behavior where Postgres/Orthanc/PACS issues are recorded but never `os.Exit`.

## RBAC + auditoría + métricas (acción como unidad transversal)
- **(2026-05-31) Decidido e implementado (en producción).** Se unifican autorización (RBAC), auditoría y métricas de uso alrededor de un único concepto: la **acción**. Cada acción transversal está declarada en un **catálogo en código** (`app/backend/cmd/server/rbac.go`, `actionCatalog`) con: `ID` estable (p. ej. `share_link.create`), `Resource`, `Permission` RBAC, `Subjects` permitidos (`patient`/`physician`/`operator`/`public`), `Mutating` (para el chequeo same-origin) y los defaults `Auditable`/`Metered`. El catálogo es la fuente de verdad porque los `ID` viajan a los logs y a la tabla `usage_events`, así que deben permanecer estables una vez publicados.
  - **Decorador `App.action(id, next)`.** Envuelve un handler HTTP y centraliza, en orden: (1) chequeo same-origin para acciones mutantes (reutiliza `sameOriginRequest`), (2) resolución del actor según `Subjects` (reusa `requirePatientSessionSummary`/`requirePhysicianSessionSummary`; *fail-closed* para acciones protegidas, actor `public` sintético solo si la acción lo permite), (3) RBAC (`role` del actor vs `Permission`), (4) emisión de evento de auditoría y (5) registro de métrica de uso. El actor resuelto se deja en el `context` (`actionScope`) para que el handler lo reuse sin un segundo lookup de sesión, y el handler puede enriquecer el registro con dimensiones propias vía `setActionDim` (p. ej. `study_uid`, `channel`).
  - **RBAC simple, sin IdP.** Los roles y sus permisos viven en código como defaults (`defaultRolePermissions`) y se pueden **sobrescribir desde `config.json`** (`rbac.roles`). No hay integración de roles con LDAP/Keycloak (no tenemos control sobre el LDAP): la única elevación soportada hoy es otorgar permiso de compartir a profesionales puntuales mediante `rbac.physician_sharers` (lista de usernames/DNIs, case-insensitive), que los mapea al rol `physician_sharer`. Defaults: `patient` puede `share_link:create`/`revoke` y `study:retrieve`; `physician` solo `study:retrieve`; `physician_sharer` agrega compartir; `operator` = `*`. Roles no presentes en config conservan sus defaults, de modo que **no se rompe el comportamiento actual** (el paciente sigue pudiendo compartir aunque no exista sección `rbac`). El endpoint de compartir para profesional ya está implementado (ver sección "Compartir estudio del profesional"); el rol y permiso lo habilitan vía `rbac.physician_sharers`.
  - **Auditoría = stdout + `channel:audit`.** `App.audit(...)` emite JSON estructurado por stdout con `"channel":"audit"` y `"msg":"audit_event"`, reutilizando el pipeline existente; el streamer `portal-logs-audit.service` lo deriva a `app/logs/audit.log`. Esquema del evento: `action`, `outcome` (`success`/`denied`/`failure`), `status_code`, `actor_kind`/`actor_id`/`actor_role`/`actor_name`, `resource`, `method`, `path`, `client_ip`, `latency_ms` + dimensiones de la acción. Los rechazos (sin sesión / sin permiso) también se auditan (`outcome:denied`, `reason`).
  - **PHI: por ahora se loguea todo.** Se difiere la minimización de PHI. El catálogo deja el hook para una política `PHIFields` por acción (lista explícita declarada, redactar el resto) cuando se trabaje; hoy el foco es trazar completo el flujo de roles/acciones.
  - **Métricas de uso = Opción B (PostgreSQL).** Nueva tabla **append-only** `usage_events` (migración `migrations/010_add_usage_events.sql`), desacoplada de las tablas operativas: `action, actor_kind, actor_id, actor_role, outcome, status_code, latency_ms, dims jsonb, occurred_at` + índices por acción/tiempo, actor y fecha. El sink se abstrae detrás de la interfaz `UsageRecorder`; la impl productiva es `PostgresUsageRecorder` (insert simple); en tests se usa un fake en memoria (sin agregar dependencias tipo `go-sqlmock`, coherente con los *interface seams* de Tier 2). No se derivan métricas de los logs (volátiles).
  - **Config.** `ExternalConfig` se extiende con `rbac` (`roles`, `physician_sharers`, `physician_default_role`) y `observability` (`audit_enabled`, `metrics_enabled`, `audit_actions`, `metered_actions`). Switches maestros (`*_enabled`) y allowlists explícitas que sobrescriben los defaults del catálogo por acción. Ejemplo en `app/config/config.example.json`. Si una sección falta, aplican los defaults del código.
  - **Alcance actual (en producción).** El decorador corre en el flujo productivo y está enganchado en las acciones reales: `patient.login` y `physician.login` (`/api/patient/login`, `/api/physician/login`; auditan identificador ingresado + resultado + IP, útil para detectar fuerza bruta), `viewer_access.grant` (`/api/{patient,physician}/studies/.../access`), `study.retrieve` (`/api/{patient,physician}/retrieve`), `study.download` (`/api/{patient,physician}/download`), `share_link.create` (`/api/{patient,physician}/studies/.../share`) y `share_link.consume` (`/share`) y `usage.read` (`/api/operator/usage/{summary,events}`, solo-lectura, operador). `share_link.revoke` queda declarada y lista (todavía no hay endpoint de revocación). El roll-out es de bajo riesgo: los handlers ya exigían sesión donde corresponde (el decorador resuelve el mismo actor), los RBAC defaults permiten todos los flujos vigentes, la auditoría/métrica es aditiva y toggleable por config, y los logins además reemplazan el `withBrowserOriginCheck` por el chequeo same-origin equivalente del decorador. Adopción de nuevas acciones = envolver el handler con `App.action(...)`.
  - **Tests.** `app/backend/cmd/server/rbac_test.go` cubre: defaults y overrides de RBAC, `hasPermission` (wildcard/denegación/permiso vacío), `roleForPhysician` (por username/DNI/default), toggles de auditoría/métricas (default/master-off/allowlist), `outcomeForStatus`, y el decorador end-to-end vía `httptest`: éxito público (métrica+auditoría+propagación de dims), denegación sin sesión (`denied`/`unauthenticated`), denegación por RBAC 403 (`denied`/`forbidden`, vía acción de catálogo temporal), rechazo cross-origin 403, apagado real por toggles de observabilidad (sin auditoría ni métrica), mapeo `outcome=failure` en 5xx, y que un error del `UsageRecorder` no altera la respuesta (se loguea). Además helpers de contexto y *nil-safety* del recorder. Verde con `make test-backend` y `go vet`.

## Fix: resolución de nodo de origen en retrieve del profesional
- **(2026-05-31) Implementado.** Síntoma: `POST /api/physician/retrieve` devolvía `502` con `source node not resolved for physician study` (visible además como `audit_event` `study.retrieve`/`outcome:failure`/`status_code:502`). Causa: `queuePhysicianRetrieve` no recibía el nodo y lo **infería** (`getPhysicianSourceNode`) desde `physician_recent_queries`/`cached_studies`; con varios nodos PACS configurados (no aplica el fallback de nodo único) y sin un match en esas fuentes (query expirada a 7 días, resultados pisados por el enriquecimiento async, o estudio ausente), no se podía resolver el origen. Decisión: el **frontend envía el nodo de origen** (`source_node_id`, que la UI ya conoce por el `source` activo / `result.source_node_id`) en el body del retrieve; el backend lo usa como **autoritativo si valida contra un nodo configurado** (`preferredConfiguredNodeID`) y **cae a la inferencia anterior** si viene vacío/ inválido (con `warn` `physician_retrieve_source_node_invalid`). El retrieve de paciente no necesita esto porque ya resuelve el origen de forma confiable desde `patient_study_access.source_json`. Tests: `physician_retrieve_test.go` (`preferredConfiguredNodeID` válido/ inválido/ vacío/ trim/ case-sensitive y `resolvePhysicianRetrieveSourceNode` con preferido válido cortocircuitando antes de la DB). Frontend: `triggerPhysicianRetrieve` adjunta `source_node_id` y los botones de recuperar llevan `data-physician-source-node`.

## Compartir estudio del profesional (gated por RBAC)
- **(2026-05-31) Implementado.** Hasta ahora compartir era solo del paciente; en el profesional únicamente existía la entrada de catálogo `share_link.create` (subject `physician`) pero sin endpoint ni botón, por eso "el frontend no mostraba la acción". Decisión: implementarlo end-to-end reusando la maquinaria existente, sin nuevo modelo de datos.
  - **Backend.** Nueva ruta `POST /api/physician/studies/{uid}/share` en `handlePhysicianStudyRoute`, envuelta en `a.action(ActionShareLinkCreate, ...)`: el decorador resuelve al médico y exige `share_link:create` (sharer → 200; profesional común → 403; todo auditado/medido como cualquier acción). `handlePhysicianStudyShare` espeja a `handlePatientStudyShare` con dos diferencias: (1) el link se persiste con **owner NULL** (`createStudyShareLink` con `patientID=""` → `study_share_links.patient_id` NULL, columna ya nullable), porque no hay fila `patients` dueña; (2) valida que el estudio esté **cacheado localmente** vía `findOrthancStudy` (mismo criterio que el viewer del profesional: solo se comparte lo ya recuperado). Devuelve QR + WhatsApp + mailto y loguea `physician_study_share_created`. El consumo del link (`/share`) no cambia: ya valida solo el snapshot del link + estudio en Orthanc, sin depender de `patient_id`.
  - **Capacidad expuesta al frontend.** `PhysicianResultsResponse` suma `can_share`, derivado de `physicianCanShare()` (`rbac.hasPermission(roleForPhysician(...), share_link:create)`). El frontend dibuja "Compartir" por estudio **solo si `can_share && result.viewer_url`** (i.e. el médico es sharer y el estudio está local), evitando mostrar un botón que daría 403. `createPhysicianStudyShare()` postea al endpoint y reusa el overlay QR del paciente (`openPatientShareQR`).
  - **Fix consumo: link sin owner (`patient_id` NULL).** Al consumir un share creado por el profesional, `studyShareLinkByToken` (`orthanc.go`) hacía `SELECT patient_id::text` y lo escaneaba en un `string` no-nullable → con `patient_id` NULL fallaba con *"converting NULL to string is unsupported"* y devolvía `500` (`audit_event` `share_link.consume`/`outcome:failure`/`status_code:500`, *"failed to load share link"*); el share del paciente nunca lo disparaba porque siempre tiene owner. Fix: `COALESCE(patient_id::text, '')`. `snapshot.PatientID` no se usa en el flujo de consumo (redirect y validación de Orthanc usan `ViewerKind`/`StudyInstanceUID`), así que `''` es seguro. Es el único punto de lectura del link, compartido por la landing `/share` y la validación de tokens de Orthanc.
  - **Tests.** `physician_share_test.go`: `TestPhysicianCanShare` (sharer por username/DNI → `true`, no-sharer → `false`, policy nil → `false`) y `TestPhysicianShareOwnerlessLinkConsumable` (regresión del consumo: un snapshot con `PatientID:""` valida como consumible, redirige al visor con el token y otorga acceso study-level en Orthanc por match de UID, sin depender del owner). La autorización 200/403 del decorador ya está cubierta genéricamente en `rbac_test.go`; el handler de creación y el scan SQL real (DB + Orthanc) son Tier 4 (diferido por decisión). Verde con `gofmt`/`go vet`/`go build`/`go test`.

## Fix: período por defecto del buscador del profesional
- **(2026-05-31) Implementado.** Síntoma: al entrar como profesional y buscar en un PACS remoto (p. ej. `hpn`) sin tocar el selector de fecha, el frontend rechazaba con *"Seleccione al menos un filtro adicional antes de consultar un PACS remoto."*; cambiando a "Esta semana" funcionaba. Causa: el `<select>` de período del profesional arrancaba en `Todos` (`value=""`, sin filtro) y `physicianDateFilter.from/to` arrancaban vacíos; como una consulta remota exige ≥1 filtro (`hasPhysicianQueryFilters`), el primer intento fallaba; el `change` a otra opción recién poblaba el rango. El paciente no tenía el problema (su selector viene con `Este mes` `selected`). Decisión: el período del profesional arranca en **"Hoy"** (`<option value="today" selected>`) e inicializar `physicianDateFilter` con `patientDateRangeForPeriod("today")` en la carga, de modo que la búsqueda remota funcione al primer intento. La restauración de estado guardado sigue pisando las fechas solo si existían (sin regresión). Cambio solo de frontend (`index.html` + `portal.js`); el repo no tiene runner de JS, la lógica pura (`patientDateRangeForPeriod`) no cambió.

En `download` la modalidad sale del estudio resuelto; en `retrieve` el estudio aún no está local, así que la modalidad la envía el frontend (campo `modality` del request, tomado de `result.modalities` de la búsqueda). Multi-modalidad se normaliza (upper, dedup, **ordenada** vía `sort.Strings`) y se une con `/`, vacío = `UNKNOWN`. Ambos desgloses usan el helper común `queryModalityBreakdown(ctx, since, action)` y cuentan solo `outcome='success'`.
      - **Backfill de `UNKNOWN` históricos** (`migrations/011_backfill_usage_modality.sql`): los eventos previos al dim se rellenan desde `qido_study_cache.modalities_json` haciendo join por `dims->>'study_uid'` (que `download`/`retrieve` ya registran). Es idempotente (solo toca filas con modalidad faltante/`UNKNOWN`) y re-ejecutable a mano con `psql -f` a medida que el cache se llena. Limitación: estudios ausentes del cache (nunca consultados por QIDO o evictados) quedan en `UNKNOWN` y no son recuperables. La normalización SQL (upper/dedup/sort) coincide con `usageModalityDim` para que los buckets sean consistentes.
      - **Top 10 profesionales por retrieves** (`top_physician_retrievers`, por DNI/usuario): se agregó la dimensión `physician_dni` (= `physician.Username`, fallback `DNI`) a los eventos `study.retrieve` del profesional. La query agrupa por `dims->>'physician_dni'` filtrando `action='study.retrieve' AND actor_kind='physician' AND COALESCE(dims->>'physician_dni','')<>''` (se omiten los retrieves previos al dim para no mezclar el ID interno/GUID del actor con los DNIs; solo aparecen DNIs reales).
    - `GET /api/operator/usage/events?window=&action=&outcome=&q=&limit=` → últimas N filas (limit default 100, máx 500), filtros opcionales por acción/outcome y **búsqueda libre `q` dentro de `dims`** (contains case-insensitive vía `position(lower($q) in lower(dims::text)) > 0`; todo parametrizado, sin inyección ni interpretación de comodines). El frontend la dispara con debounce de 300 ms.

Cambio de frontend en `index.html`/`portal.js`/`portal.css`.

## Email del código de acceso: `pacienteApp` como fuente principal
- **(2026-07-02) Implementado.** Hasta ahora el email destinatario del código de acceso (`POST /api/patient/send-code`, modo `mail`) se resolvía **solo** desde el primer contacto activo de tipo `email` en `paciente.contacto`. Decisión: la fuente **principal** pasa a ser la cuenta de la colección Mongo `pacienteApp` y `paciente.contacto` queda como **fallback**.
  - **Resolución (`identity.go`, `his_mongo_direct`).** Tras resolver el documento en `paciente`, `MongoPatientIdentitySource.ResolveByDocument` consulta `pacienteApp` con `$elemMatch` sobre `pacientes` (`{id: <paciente._id>, relacion: "principal"}`, projection `{_id, email}`) y toma el `email` de la **raíz** del documento `pacienteApp`. La precedencia la encapsula la función pura `preferPatientAppEmail(appEmail, contactoEmail)` (app gana si no está vacío; si no, cae al contacto vía `activePatientContactoEmail`). El resto del flujo (`handlePatientSendCode`, ofuscación, `missing_active_email`) no cambia: solo cambia de dónde sale `identity.Email`.
  - **Best-effort, no bloqueante.** La consulta a `pacienteApp` es read-only y su error **no** interrumpe el login: se loguea `paciente_app_email_lookup_failed` (warn) y se cae al email de contacto. `mongo.ErrNoDocuments` (sin cuenta principal) se trata como "sin app email", no como error.
  - **Base para impersonación futura.** El vínculo cuenta↔`pacientes[]` (con `relacion`) es también la base prevista para que, más adelante, una cuenta pueda impersonar perfiles relacionados (p. ej. hijos) en el portal; por eso se modela `pacientes[].relacion` aunque hoy solo se use `principal`.
  - **Tests.** `identity_test.go`: `TestActivePatientContactoEmail`, `TestPreferPatientAppEmail` y `TestMongoPacienteToPatientIdentity_EmailPrecedence` (Tier 1 puro). La consulta Mongo real (`resolvePrincipalAppEmail`) requiere DB → Tier 4 (diferido por decisión, consistente con el resto del backend). Verde con `gofmt`/`go vet`/`go test` vía `make test-backend`.

## Fix: retrieve incompleto — detección (`local_partial`) + remediación
- **(2026-07-03) Implementado (detección).** Síntoma reportado: algunas tomografías (CT) traídas desde Synapse (`hpnsyn`) abren en el visor pero **algunas series dan 403**. Causa distinta al 403 del buscador: el retrieve se corta (Synapse inestable — C-ECHO `context deadline exceeded` de forma sostenida en el log — y timeout duro de 15 min del worker), así que **faltan series**. El código marcaba `local_complete` con solo dos condiciones: el job C-GET de Orthanc reporta `Success` y el estudio "aparece" en Orthanc (`waitForStudyInOrthanc`); **nunca verificaba completitud** (ni conteo de series/instancias, ni sub-operaciones fallidas del job). Cuando el visor pide una serie que no llegó, el plugin de autorización responde `403 Unknown resource`.
  - **Decisión de diseño (acordada con el usuario).** (1) Verificar completitud combinando **series + instancias** (a nivel serie, y dentro de cada serie el conteo de instancias). (2) Reintentar **solo lo faltante**, sin retrabajar lo ya traído (C-GET `Level: "Series"` con las `SeriesInstanceUID` faltantes). (3) Reintentos **configurables en `config.json`**, empezando en 3, con backoff. (4) **No bloquear** la visualización ni otras operaciones sobre un estudio incompleto.
  - **Alcance de este PR (detección).** Nuevo estado `local_partial` (además de `local_complete`/`not_local`). Tras un C-GET exitoso, `verifyRetrievedStudyCompleteness` consulta al origen las series esperadas (`NumberOfSeriesRelatedInstances` vía C-FIND a nivel serie para nodos `c_find`, o QIDO `/series` para `qido_rs`) y a Orthanc local las series presentes (vía `POST /tools/find` Series+Expand, contando `Instances`), y compara con `evaluateStudyCompleteness` (puro). Si el origen no da conteos confiables, la evaluación se **degrada a best-effort** (no se afirma `local_partial`, se mantiene el comportamiento previo). Migración `013` agrega a `cached_studies`: `expected_series_count`, `present_series_count`, `missing_series_json`, `last_completeness_checked_at`. `getStudyOperationalState` deja de enmascarar `local_partial` como `local_complete` pero **mantiene las URLs de visor** (sigue siendo abrible). La completitud es best-effort: un error de lookup no falla el retrieve, solo impide afirmar `local_partial`.
  - **Remediación (incluida en el mismo PR).** Si el estudio queda `local_partial`, `remediateIncompleteRetrieve` reintenta **solo las series faltantes** vía C-GET `Level: "Series"` (`startOrthancCGetSeries`, Resources por `SeriesInstanceUID`), sin retrabajar lo ya traído, re-verificando completitud entre intentos y cortando apenas queda completo. Configurable en `config.json`/`portal`: `retrieve_max_attempts` (default 3, total = 1 inicial + reintentos), `retrieve_retry_backoff_seconds` (default 10, backoff lineal creciente para no saturar un origen inestable como Synapse) y `retrieve_timeout_minutes` (default 30, reemplaza el timeout duro de 15 min del worker que era una causa directa de cortes en CT grandes). Los reintentos actualizan el job a fase `completing` (UI: "Completando series faltantes"). Un estudio parcial sigue siendo `available_local`/visualizable durante la remediación. Los fallos de reintento se loguean y se toleran (se persiste la completitud alcanzada).
  - **Tests.** `completeness_test.go` (Tier 1, puro) cubre `evaluateStudyCompleteness` (completo, serie ausente, serie corta, conteo desconocido = solo presencia, best-effort sin conteos, orden determinista de `MissingSeries`) y `completionPercent` (prioriza instancias, cae a series, clamp 0..100). Las consultas reales a origen/Orthanc y el loop de C-GET son Tier 4 (diferido). Verde con `make test-backend`.
- **(2026-07-18) Corregido: el "best-effort" seguía marcando estudios de Synapse como `local_complete` siendo parciales.** Reincidencia del síntoma original: el backend afirmaba completo cuando no lo era. Tres causas encadenadas, todas específicas de `hpnsyn`/Synapse: (1) `verifyRetrievedStudyCompleteness` obtiene las series esperadas con **otro C-FIND a Synapse**; si ese C-FIND falla o vuelve vacío —justo cuando Synapse está inestable, la misma condición que dejó el retrieve incompleto— el reporte quedaba `Evaluated=false` y `completeRetrieveSuccess` caía a `local_complete`. (3) El job C-GET de Orthanc se tomaba como éxito con sólo `State == "Success"`, **sin mirar `FailedInstancesCount`/`RemainingInstancesCount`** del `Content`.
  - **Decisión de diseño.** Nunca afirmar `local_complete` sobre datos no verificados. Nuevo estado `local_unverified` (además de `local_complete`/`local_partial`/`not_local`): estudio presente y visualizable, pero cuya completitud no pudo confirmarse contra el origen. Configurable con `portal.retrieve_verify_instance_counts` (default `true`). `getStudyOperationalState` preserva también `local_unverified` (no lo enmascara como completo) y mantiene las URLs de visor. ## Fix: búsqueda local del profesional vía `POST /tools/find` (no QIDO)
- **(2026-07-03) Implementado.** Síntoma: al buscar como profesional por paciente en el caché local, `GET /api/physician/results?...&patient_id=NNN&source=local_cache` devolvía **500** (para todos los pacientes: 8025031, 4671724, 34299314, 33165830, 94913308), mientras que la misma búsqueda **sin** `patient_id` (solo fecha o `patient_name`) daba **200**, y el mismo `patient_id` contra un PACS remoto (`hc`/`hpn`/`hpnsyn`) daba **200**.   - **Causa.** `searchPhysicianResultsFromLocalCache` (`physician.go`) consultaba el Orthanc local por **QIDO `GET /dicom-web/studies?PatientID=...`**. Con el plugin de autorización en `CheckedLevel: studies`, una QIDO filtrada por `PatientID` se clasifica como acceso a un recurso concreto; si ese paciente aún no tiene estudios cacheados localmente, el plugin responde `Unknown resource` → **403**, y el backend lo propaga como **500**. Las búsquedas sin identificador de recurso (solo fecha/nombre) se tratan como búsqueda genérica y pasan. Esto es exactamente lo que ya advertía la decisión previa: *"backend local-study detection must not depend on internal QIDO `GET /dicom-web/studies?...` calls ... backend local lookup now relies on `POST /tools/find`"*. Ese path del profesional nunca se había migrado (a diferencia de `findOrthancStudy`, que ya usaba `/tools/find`).
  - **Fix.** Migrar la búsqueda de caché local del profesional a **`POST /tools/find`** (`Level: "Study"`, `Query` con `PatientID`/`PatientName`/`ModalitiesInStudy`/`StudyDate`, `Expand: true`, `CaseSensitive: false`, `Limit: 200`, `RequestedTags: ["ModalitiesInStudy","NumberOfStudyRelatedInstances"]`). Se autoriza por el token interno (`X-Orthanc-Internal-Token` → perfil con permiso `all`), igual que el resto del tráfico interno. El mapeo a `PhysicianResult` toma `MainDicomTags`/`PatientMainDicomTags` y deriva modalidades y cantidad de imágenes desde `RequestedTags`. El resto del flujo (estado operacional, `locations`, overlay de cache persistido, enriquecimiento ANDES, orden) no cambia.
